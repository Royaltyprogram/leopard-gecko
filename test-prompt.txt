FocusBoard라는 이름의 Next.js App Router + TypeScript + Tailwind 프로젝트를 초기화하고, 불필요한 기본 예제 코드는 제거해줘. 첫 화면은 빈 랜딩 페이지로 두고 실행 방법을 README에 간단히 적어줘.

폴더 구조를 정리해줘. src/app, src/components, src/lib, src/server, src/features, src/styles 기준으로 재구성하고 path alias도 깔끔하게 맞춰줘.

ESLint/Prettier 설정을 단순하게 정리하고, import 정렬과 기본 코드 스타일이 일관되게 유지되도록 설정해줘. 과한 룰은 넣지 말아줘.

환경변수 구조를 잡아줘. .env.example을 만들고, 서버 시작 시 필요한 값이 없으면 명확하게 실패하도록 간단한 env 검증 유틸을 추가해줘.

Prisma를 붙이고 SQLite를 사용하도록 설정해줘. 우선 User, Workspace, Board, Column, Task 모델의 초안을 설계해줘.

Prisma 스키마를 보완해서 관계와 정렬 순서용 필드를 명확히 넣어줘. 각 모델 이름과 필드는 최대한 직관적으로 유지해줘.

첫 마이그레이션을 만들고, 로컬에서 바로 볼 수 있는 샘플 데이터가 들어가도록 seed 스크립트를 추가해줘. 사용자 2명, 워크스페이스 1개, 보드 2개 정도면 충분해.

서버에서 Prisma 클라이언트를 안전하게 재사용하는 유틸을 추가해줘. 개발 모드에서 중복 인스턴스 문제가 없게 단순하게 처리해줘.

src/server 아래에 데이터 접근 레이어를 만들어줘. 우선 보드 목록 조회, 보드 상세 조회, 컬럼 생성, 태스크 생성 함수만 분리해줘.

전체 앱의 기본 레이아웃을 만들어줘. 상단 헤더, 좌측 사이드바, 메인 콘텐츠 영역 정도의 단순한 앱 셸이면 충분해.

랜딩 페이지를 만들어줘. 제품 이름, 한 줄 설명, 데모 앱으로 들어가는 버튼 정도만 있는 깔끔한 화면이면 된다.

데모용 인증 방식을 아주 단순하게 넣어줘. 실제 회원가입 대신 seed된 사용자 중 하나로 로그인하는 세션 기반 구조를 구현해줘.

로그인 페이지를 만들고, seed된 두 사용자 중 하나를 선택해서 로그인할 수 있게 해줘. 디자인은 단순하고 읽기 쉽게 해줘.

보호된 라우트 구조를 잡아줘. 로그인하지 않으면 앱 영역으로 못 들어가고 로그인 페이지로 보내지게 해줘.

로그인 후 진입하는 대시보드 페이지를 만들어줘. 최근 보드, 오늘 마감 태스크 수, 진행 중 태스크 수 정도만 간단히 보여줘.

좌측 사이드바에 워크스페이스 정보와 보드 목록을 표시해줘. 현재 선택된 보드는 명확히 강조해줘.

보드 목록 화면을 만들고, 보드 카드 UI를 추가해줘. 이름, 설명, 태스크 개수 정도만 보여주면 된다.

보드 생성 기능을 추가해줘. 모달 또는 인라인 폼 중 더 단순한 쪽으로 구현하고, 생성 후 목록이 즉시 갱신되게 해줘.

보드 수정과 삭제 기능을 추가해줘. 삭제는 확인 단계를 한 번 거치게 하고, 코드 흐름은 복잡하지 않게 유지해줘.

보드 상세 페이지의 기본 골격을 만들어줘. 컬럼들이 가로로 나열되는 칸반 레이아웃만 우선 잡아줘.

컬럼 생성 기능을 추가해줘. 보드 상세 페이지에서 새 컬럼을 만들 수 있어야 하고, 생성 후 바로 보이게 해줘.

컬럼 이름 수정과 컬럼 삭제 기능을 추가해줘. 삭제 시 해당 컬럼의 태스크 처리 방식은 우선 “삭제 불가”로 막고 경고만 보여줘.

태스크 카드 컴포넌트를 만들어줘. 제목, 우선순위, 마감일 정도만 카드에 보이게 하고 UI는 과하지 않게 해줘.

각 컬럼에서 태스크를 생성할 수 있게 해줘. 최소 입력값은 제목만 필요하게 하고, 나머지는 나중에 확장 가능하도록 단순하게 설계해줘.

태스크 상세를 우측 드로어 또는 모달로 열 수 있게 해줘. 제목, 설명, 상태, 마감일을 수정할 수 있게 해줘.

태스크 설명 필드를 추가하고, 여러 줄 텍스트 입력이 가능하게 해줘. 너무 무거운 에디터 말고 기본 textarea로 충분해.

태스크에 우선순위(low, medium, high)를 추가해줘. 저장과 표시가 모두 되게 하고 색상 구분은 은은하게만 넣어줘.

태스크에 마감일 필드를 추가해줘. 카드와 상세 화면 모두에서 일관되게 보여주고, 날짜 포맷 유틸도 분리해줘.

태스크에 담당자 필드를 추가해줘. 현재 워크스페이스 사용자 중 한 명을 선택할 수 있게 해줘.

태스크 카드에 담당자 아바타 대신 이니셜 배지를 보여줘. 과한 아바타 시스템은 만들지 말고 단순하게 구현해줘.

태스크를 같은 컬럼 안에서 드래그 앤 드롭으로 순서 변경할 수 있게 해줘. 구현은 안정성과 가독성을 우선해줘.

태스크를 다른 컬럼으로 이동할 수 있게 해줘. 이동 후 컬럼과 순서가 DB에 저장되게 해줘.

컬럼 자체도 좌우로 순서 변경할 수 있게 해줘. 정렬용 필드 업데이트 로직을 읽기 쉽게 정리해줘.

드래그 앤 드롭 이후 낙관적 UI 업데이트를 넣어줘. 실패 시 롤백되도록 하되, 코드가 지나치게 복잡해지지 않게 해줘.

태스크 라벨 기능을 추가해줘. Label 모델을 만들고, 색상과 이름만 가지는 단순한 구조로 시작해줘.

보드 단위 라벨 관리 UI를 만들어줘. 라벨 생성, 이름 수정, 색상 수정, 삭제가 가능해야 해.

태스크 상세 화면에서 라벨을 여러 개 선택할 수 있게 해줘. 선택 결과가 카드에도 작게 보이게 해줘.

보드 상세 상단에 필터 바를 추가해줘. 담당자, 우선순위, 라벨로 필터링할 수 있게 해줘.

필터 상태를 URL 쿼리스트링과 동기화해줘. 새로고침해도 같은 필터 상태가 유지되게 해줘.

태스크 제목 검색 기능을 추가해줘. 디바운스는 과하게 넣지 말고, 입력 반응이 자연스러운 정도만 적용해줘.

“내 태스크” 페이지를 추가해줘. 현재 로그인한 사용자가 담당자인 태스크만 모아서 보이게 해줘.

“오늘 해야 할 일” 위젯을 대시보드에 추가해줘. 오늘 마감이거나 이미 지난 태스크를 묶어서 보여줘.

태스크 활동 로그 모델을 추가해줘. 생성, 제목 수정, 상태 이동, 담당자 변경 정도만 기록해줘.

활동 로그를 남기는 서버 유틸을 만들어줘. 태스크 관련 주요 변경 지점에서 재사용할 수 있게 단순하게 분리해줘.

태스크 상세 화면 하단에 활동 타임라인을 보여줘. 최신순으로 간단하게 읽히는 형태면 충분해.

태스크 댓글 기능을 추가해줘. Comment 모델을 만들고, 작성자와 본문, 생성 시간만 우선 저장해줘.

태스크 상세 화면에 댓글 목록과 댓글 작성 폼을 추가해줘. 실시간까지는 필요 없고, 등록 후 즉시 갱신되면 된다.

댓글 수정과 삭제 기능을 추가해줘. 작성자 본인만 수정/삭제할 수 있게 단순한 권한 체크를 넣어줘.

태스크 체크리스트 기능을 추가해줘. ChecklistItem 모델을 만들고 완료 여부를 토글할 수 있게 해줘.

태스크 상세 화면에 체크리스트 섹션을 추가해줘. 항목 추가, 수정, 삭제, 완료 토글이 가능해야 한다.

체크리스트 진행률을 태스크 카드에 작게 표시해줘. 예를 들면 2/5 형태면 충분하다.

노트 기능을 추가해줘. Note 모델을 만들고 워크스페이스 단위로 제목과 본문을 저장할 수 있게 해줘.

노트 목록 페이지를 추가해줘. 좌측 리스트, 우측 본문 구조의 단순한 노트 앱처럼 만들어줘.

노트 편집기는 너무 무겁지 않게 markdown textarea 기반으로 구현해줘. 우측에 간단한 미리보기를 보여줘도 좋다.

태스크와 노트를 연결할 수 있게 해줘. 태스크 상세 화면에서 관련 노트를 선택하고 연결 상태를 볼 수 있게 해줘.

달력 페이지를 추가해줘. 마감일이 있는 태스크를 월 단위로 보여주는 단순한 캘린더 뷰를 만들어줘.

달력에서 날짜를 클릭하면 그날 마감인 태스크 목록이 보이게 해줘. 화면 흐름은 최대한 단순하게 유지해줘.

태스크를 달력에서 다른 날짜로 옮겨 마감일을 바꿀 수 있게 해줘. 구현이 복잡하면 드래그 대신 클릭 기반 이동으로 단순화해도 된다.

대시보드에 보드별 태스크 수, 완료 비율 정도를 보여주는 간단한 통계 섹션을 추가해줘.

통계 UI는 차트 라이브러리 없이도 괜찮다. 우선은 막대형 스타일의 단순한 시각화 컴포넌트로 구현해줘.

전체 앱의 로딩 상태를 다듬어줘. 페이지 진입, 목록 로딩, 저장 중 상태에 대해 skeleton 또는 작은 상태 표시를 넣어줘.

에러 처리 방식을 정리해줘. 서버 액션이나 API 호출 실패 시 사용자에게 읽기 쉬운 메시지를 보여주고, 내부 에러는 로깅만 하게 해줘.

폼 검증을 Zod 기반으로 정리해줘. 보드 생성, 태스크 생성, 댓글 생성, 노트 생성/수정에 우선 적용해줘.

공통 폼 컴포넌트를 너무 많이 만들지 말고, 반복이 분명한 부분만 작게 추출해서 코드 중복을 줄여줘.

키보드 단축키를 몇 개 추가해줘. 예를 들면 /는 검색 포커스, n은 새 태스크, g d는 대시보드 이동 정도면 충분하다.

커맨드 팔레트 UI를 추가해줘. 보드 이동, 태스크 검색, 새 태스크 생성 같은 기본 액션을 빠르게 실행할 수 있게 해줘.

빠른 태스크 생성 패널을 만들어줘. 제목만 입력하고 현재 보드의 특정 컬럼에 바로 생성할 수 있게 해줘.

모바일 레이아웃을 정리해줘. 보드 화면이 작은 화면에서도 무너지지 않도록 가로 스크롤과 여백을 다듬어줘.

모바일에서 태스크 상세 화면이 잘 보이도록 드로어/모달 동작을 조정해줘. 터치 환경에서 너무 답답하지 않게 해줘.

접근성 점검을 해줘. 버튼 라벨, 폼 레이블, 키보드 포커스, 색 대비에서 큰 문제를 먼저 해결해줘.

빈 상태 화면들을 추가해줘. 보드가 없을 때, 태스크가 없을 때, 검색 결과가 없을 때 각각 어색하지 않게 보여줘.

404 페이지와 일반 에러 페이지를 프로젝트 분위기에 맞게 간단히 만들어줘. 디자인보다 명확한 안내를 우선해줘.

설정 페이지를 추가해줘. 사용자 이름 표시, 현재 워크스페이스 정보, 기본 보드 선택 정도만 편집 가능하게 해줘.

보드 기본 설정을 추가해줘. 새 태스크 기본 우선순위, 기본 담당자 정도를 보드 설정에서 바꿀 수 있게 해줘.

JSON 내보내기 기능을 추가해줘. 현재 워크스페이스의 보드/컬럼/태스크/댓글/노트를 하나의 JSON으로 다운로드할 수 있게 해줘.

JSON 가져오기 기능을 추가해줘. 기존 데이터를 덮어쓰지 말고 새 워크스페이스로 가져오는 단순한 방식으로 구현해줘.

테스트 환경을 잡아줘. Vitest와 Testing Library를 설정하고, 최소한 렌더링 테스트가 돌아가게 만들어줘.

유틸 함수 테스트를 추가해줘. 날짜 포맷, 정렬 계산, 필터링 함수처럼 순수 함수 위주로 먼저 테스트해줘.

서버 데이터 레이어 테스트를 추가해줘. 보드 조회, 태스크 생성, 태스크 이동 로직을 우선 검증해줘.

핵심 UI 컴포넌트 테스트를 추가해줘. 태스크 카드, 필터 바, 보드 리스트 정도를 대상으로 해줘.

Playwright를 붙이고 기본 E2E 테스트를 추가해줘. 로그인 후 대시보드 진입이 성공하는 시나리오부터 만들어줘.

E2E로 보드 생성 시나리오를 추가해줘. 생성 후 목록에 바로 보이는지까지 검증해줘.

E2E로 태스크 생성/수정 시나리오를 추가해줘. 제목과 마감일 수정이 정상 반영되는지 확인해줘.

E2E로 태스크 이동 시나리오를 추가해줘. 컬럼 간 이동 후 새로고침해도 위치가 유지되는지 검증해줘.

전체 코드에서 중복되는 서버 응답 처리와 에러 메시지 구성을 정리해줘. 너무 일반화하지 말고 반복이 확실한 것만 묶어줘.

컴포넌트 구조를 한 번 정리해줘. 지나치게 큰 파일을 몇 개 골라 작은 단위로 분리하되, 추상화는 최소화해줘.

타입 정리를 해줘. Prisma 타입, UI용 뷰모델 타입, 폼 입력 타입이 섞여 있다면 읽기 쉽게 경계를 나눠줘.

상수와 매직 스트링을 정리해줘. 우선순위 값, 라벨 색상 후보, 기본 에러 문구 같은 것만 공통 상수로 빼줘.

로깅 유틸을 추가해줘. 개발 환경에서는 보기 쉽고, 프로덕션에서는 과도한 개인정보가 찍히지 않도록 단순하게 처리해줘.

성능을 한 번 점검해서 큰 낭비가 있는 부분만 개선해줘. 예를 들면 불필요한 전체 리렌더링이나 중복 조회가 있으면 정리해줘.

보드 화면에서 데이터 갱신 흐름을 정리해줘. 생성/수정/삭제 후 새로고침 없이 자연스럽게 반영되도록 일관성을 맞춰줘.

seed 데이터를 더 풍부하게 만들어줘. 완료된 태스크, 지연된 태스크, 라벨 많은 태스크, 댓글 있는 태스크 등 데모하기 좋은 데이터로 보강해줘.

npm run db:reset 같은 개발용 명령을 추가해줘. DB 초기화, 마이그레이션, seed까지 한 번에 되게 해줘.

README를 제대로 써줘. 프로젝트 소개, 실행 방법, 주요 기능, 폴더 구조, 테스트 실행 방법을 짧고 명확하게 정리해줘.

docs/architecture.md를 추가해줘. 왜 이런 구조로 나눴는지, 데이터 흐름이 어떻게 되는지, 핵심 도메인 모델이 무엇인지 간단히 설명해줘.

docs/demo-script.md를 추가해줘. 데모할 때 5분 안에 보여줄 흐름을 순서대로 적어줘. 로그인, 보드 진입, 태스크 생성, 이동, 댓글, 노트, 달력 정도를 포함해줘.

Dockerfile을 추가해줘. 로컬 데모 용도로 충분한 수준이면 되고, 과하게 최적화하지 않아도 된다.

GitHub Actions CI를 추가해줘. 최소한 lint, typecheck, unit test, e2e 중 실행 가능한 것들을 순서 있게 돌리게 해줘.

전체 프로젝트를 최종 점검해줘. lint, typecheck, test 기준으로 깨지는 부분을 모두 수정하고, 불필요한 죽은 코드는 제거해줘.

마지막으로 코드베이스 전반을 한 번 더 다듬어줘. 읽기 어려운 이름, 어색한 UI 문구, 불균형한 여백, 불필요한 추상화를 정리하고 “단순하지만 완성도 있는 데모용 제품” 상태로 마무리해줘.
